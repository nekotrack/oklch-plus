# OKLCH グラデーションで意図しない色が現れる理由
## ── 色相制御を保ったまま、なめらかに解消する ──

---

あなたは OKLCH で美しいグラデーションを作ろうとした。
青から黄色へ。シアンから赤へ。

でも画面に現れたのは、意図していなかった色の帯だった。青と黄の中間に、なぜか緑が現れる。**OKLCH のグラデーションは、「諸刃の剣」なのです。**

「OKLCH を使うとパレットが作りやすい」——そういった記事をご覧になったことがある方も多いと思います。L を一定にして H を回転させるだけで視覚的に揃ったパレットが作れる。それは本当のことです。でも、グラデーション補間の落とし穴はあまり語られていません。

---

## 「きれいなグラデーション」の約束と落とし穴

OKLCH は、デザイナーに多くのことを約束してくれる色空間です。

- 明度が均一に変化する
- 色相が予測通りに動く
- 彩度の操作が直感的

HSL や RGB のような「見た目は 50% でも、感覚的には違う」というズレが少ない。だから、パレットの自動生成やアクセシビリティの確保に、急速に採用が広がっています。

でも、ひとつだけ、静かに潜む落とし穴があります。

---

## 問題の構造：色相の迂回

OKLCH の色は、3つの数値で表されます。

```
L = 明度（0〜1）
C = 彩度（0〜0.4 程度）
H = 色相（0〜360°）
```

鮮やかな緑は `L=0.7, C=0.28, H=145°` のような値を持ちます。
では、黒は？

```
L=0, C=0, H=???
```

OKLCH の色相は 0°〜360° の円環です。青（H≈262°）から黄（H≈93°）へ補間するとき、短い経路（約169°）を通れば自然ですが、長い経路（約191°）を通ると緑方向を迂回します。この迂回が意図しない色の帯を生み出します。

また、黒に色相はありません。彩度がゼロということは、色相という概念自体が意味を持たなくなる。数学的に言うと、黒は「色相が未定義の点」です。

グレーも同じです。どれだけ薄いグレーでも、完全に無彩色であれば C=0 であり、色相は存在しません。この記事では黒を代表として話を進めますが、ダークグレーへのグラデーションでも同じことが起きます。

---

## グラデーションは「2点間を結ぶ旅」

OKLCH でグラデーションを作るとき、コンピューターは 2 色の間を **L・C・H それぞれ別々に補間** します。

```
青: L=0.53, C=0.26, H=262°
黄: L=0.97, C=0.21, H=110°
```

L と C はそれぞれの値へ向かいますが、H は？ 262° から 110° へ補間するとき、最短経路（−152°）を通れば自然ですが、逆回り（+208°）を通ると緑方向（H≈180°）を経由します。

C が低下する中間域で、色相が意図しない方向を向いている——その結果、**補間経路の中間に緑っぽい帯**が現れます。これが色かぶりの正体です。

青から赤へのグラデーションでも、同じ理由で橙が混じります。

---

## なぜ今まで問題にされなかったのか

この問題は、実は色の専門家の間では知られていました。W3C（ウェブの標準を決める機関）の議論でも、2021年から取り上げられています。

でも、多くの場合は「見えにくかった」のです。

明度も彩度も低い領域は、そもそも**暗くてほぼ黒に近い色**。人間の目はそこで色の違いを識別しにくい。また、ブラウザが自動でクリッピング（色の切り捨て）をするため、問題が表示レベルで隠蔽されることもあります。

これまでの主な対処法は、「Oklab（デカルト座標の色空間）でグラデーションを作る」というものでした。Oklab では色相という概念が補間に介在しないので、問題そのものが起きない。でも、それは **OKLCH の色相制御という利点を丸ごと諦める**ことを意味します。

---

## 解決策：「地図を信じる度合い」を彩度で変える

ここで考え方を変えます。

色相環は「色の地図」です。でも、彩度がゼロに近い領域では、その地図は意味を失います。無彩色の近くでは、色相という概念自体が溶けていくのだから。

だとすれば：**彩度が高いときは色相環（地図）に従い、彩度が低くなるにつれて地図を信じる度合いを減らしていけばいい。**

具体的には、各補間ステップの彩度 C に応じて、「OKLCH 補間」と「Oklab 線形補間」をブレンドします。

```
彩度が高い → OKLCH 極座標で補間（色相制御を保つ）
彩度が低い → Oklab 線形で補間（色相の影響をゼロに）
```

そしてこのブレンドの割合を、**Naka-Rushton 関数**という S 字カーブで制御します。「中程度の彩度で最も効き、端点でゼロへ滑らかに収束する」という非線形の塩梅が、この用途にぴったり合います。

---

## 結果

黒への旅の途中で、補間は静かに判断を変えます。彩度が落ちるにつれて、色相環の案内を手放し、もっとシンプルな直線の道を選ぶ。その切り替えは連続的で滑らかなので、**どこで変わったかはわからない。ただ、グラデーションがきれいになっている。**

茶色は消え、緑は緑のまま黒へ向かいます。
紫も消え、青は青のまま暗くなっていきます。

---

## デザイナーの皆様への実務的なポイント

**影響を受けやすいケース**
- 色相差が大きい有彩色間の補間（例：青→黄、シアン→赤）
- カラースケール生成で中間ステップの彩度が低くなる経路
- UI の色状態変化（hover→active 等）で色相をまたぐ場合

**影響を受けにくいケース**
- 同じ色相内で明度だけ変える補間
- 彩度がもともと低いパレット
- 隣接色相間の補間（色相差が30°以内程度）

**ブラウザの CSS でできることとの違い**
モダンブラウザの CSS は無彩色端点への色相継承で有彩色→黒の問題を部分的に解消しています。しかし**有彩色間の補間経路での色かぶりは CSS でも残ります**。本記事の手法は、JavaScript ベースのカラーエンジンや、独自の補間実装を持つデザインツールで取り入れる余地があります。

---

## おわりに

OKLCH は素晴らしい色空間です。その「色相を直感的に操れる」という強みは本物で、デザインシステムの構築やアクセシビリティの確保に大きな力を発揮します。

今回の問題は、その強みの裏側にある構造的な弱点でした。でも、弱点があることより、**その弱点を連続的に・追加パラメータなしで解消できた**と考えています。

グラデーションが「ただきれいになった」だけではなく、なぜそうなるかを理解することで、次に問題が起きたときに自分で考えられるようになる。そのためのヒントとして、この記事が役立てば幸いです。

## 補足：Naka-Rushton 関数とは

この解法で使った「S 字カーブ」には、ちゃんと名前と由来があります。

**中 研一（Kenichi Naka）** と **William A.H. Rushton** ── 二人の生理学者が 1966 年に発表した研究に基づく関数です。

中氏は日本の神経生理学者で、テンチ（Tinca tinca・コイ科の淡水魚）の網膜を使った実験から光応答の特性を解明しました。Rushton 氏はイギリスの生理学者で、視覚色素と暗順応の研究で知られます。二人が記述したのは、「目の光受容体（桿体・錐体）が光の強さにどう反応するか」という生物学的な圧縮特性でした。

弱い光には敏感に反応し、強い光に対しては徐々に飽和していく。その滑らかな S 字の形が、まさに今回必要だった「彩度に応じた信頼度の変化」にぴったり一致していました。

もともとは網膜の話です。でも「彩度が低くなるにつれて色相を信頼しなくなっていく」という感覚は、「光が弱くなるにつれて色の識別をやめる」という人間の視覚の性質とよく似ています。生物学のモデルが、思いがけない形で色の計算に役立った。そういう偶然の一致が、この問題を解くうえで一番面白い部分だったかもしれません。
