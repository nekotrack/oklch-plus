色選びで時間を使いすぎる、という経験はないでしょうか。

カラーピッカーを開いてみたはいいものの、なんとなくしっくりこない。
変えるたびにバランスが崩れる。結局どれがいいかわからなくなる。

色が苦手な理由のひとつは、**使っている色空間が自分の感覚と合っていない**からかもしれません。

---

## 「明るく」「鮮やかに」がそのまま動く

sRGB の Hex 値は直感的な決定が難しく、HSL の場合は L を 50% にしても、色によって見え方がバラバラで感覚と操作が一致しません。

一方、同じ LCH でも OKLCH は直感とよく合います。

- **L（Lightness）** = 明るさ
- **C（Chroma）** = 鮮やかさ
- **h（Hue）** = 色の種類

「もう少し明るく」と思ったら L を上げる。「もう少し落ち着いた色に」と思ったら C を下げる。
言葉と操作が一対一で対応します。

## 使い方は 3 ステップ

**[ツールのスクリーンショット]**

👉 [時彩](#)（リンクは後で追記）

### Step 1：情景を選ぶ

6 つの情景から選ぶと、その情景の色相（Hue）がベースになります。

| 情景 | 色の方向 |
|---|---|
| 茜・緋 | 赤橙 |
| 柿・麦 | 黄 |
| 柳・翠 | 緑 |
| 空・海 | 青 |
| 露・藍 | 青紫 |
| 藤・宵 | 紫 |

業種や用途ではなく、**自然の情景から色の方向を選ぶ**設計です。
「コーポレートだから青」ではなく、「空・海のような青」という選び方になります。

### Step 2：時間帯を選ぶ

情景を選ぶとすぐ下に、その色相で 6 つの時間帯を表示したストリップが現れます。

深夜 / 夜明け / 朝 / 正午 / 夕暮れ / 宵

それぞれ明るさ（L）と鮮やかさ（C）が異なります。夕暮れは最も彩度が高く、深夜は沈んだ暗い色になります。実際の色を見ながら時間帯をタップするだけで、パレットが自動生成されます。

### Step 3：Hue を「回して」止める

ここが個人的に気に入っている操作です。

`色相 H 回転` を押すと、色が 30°/秒でゆっくり変わり続けます。気に入ったところで止める、それだけです。

「選ぶ」というより、**色と出会う**感覚に近いです。スライダーを慎重に動かすより、回転させてから止める方が、むしろ意外な色を見つけやすいと感じています。単純に色が変わり続けるのを眺めるだけでも、なかなか楽しいです（個人の感想）。

フッターのスライダーや RESEARCH タブの L-C ガマット図上の現在値ポインターをドラッグして、L・C・h を個別に調整することもできます。

## AI・デザインツールに渡す

気に入ったパレットができたら「md を書き出し」ボタンを押します。CSS Variables で書かれた .md ファイルがダウンロードされます。

色を渡せる AI は選びません。

**Claude / Claude.ai** なら、.md をアップロードしてプロンプト例を貼り付けるだけです。**ChatGPT** にも同じ .md を渡せます。変数名で色の役割が明示されているので、指示の書き方は変わりません。**Claude Code** で開発中であれば、プロジェクトに .md を置いて参照させることで、カラーシステムを意識したコーディングを依頼できます。**Figma** は OKLCH のネイティブサポートが未実装ですが、プラグイン（OkColor 等）経由で OKLCH を扱えます。また、`:root { }` ブロックだけ取り出したい場合は「CSS をコピー」で直接取得できます。**Tailwind CSS v4 は OKLCH をネイティブ採用**しているため、設定ファイルにそのまま記述できます。`:root` を `@theme` に書き換えるだけで使えます。

用途はWebページに限りません。**プロトタイプや個人制作であればこのまま十分なケースが多く**、本番プロダクト向けには出発点として使い、必要に応じて調整するという使い方もできます。Webアプリ・コンポーネント・AIに生成させるSVGなど、AIが色の役割を読める形式であれば指示が通ります。

仕上がりのイメージは、ツール内の見本欄で確認しながら調整できます。

---

## 2 つのワークフロー

### 色を先に決めてから渡す

ツールで情景・時間帯を選んで色を決める → md を書き出す → AI・デザインツールに渡す → デザイン・コード完成。

「最初に世界観を決めたい」「ブランドカラーを起点にしたい」という場合に向いています。

### 骨格を先に作って色を後から流し込む

AI に「グレースケールで構成だけ作って」と頼んでおいて、骨格が固まったところでツールを開いて色を選ぶ。できた .md を渡して「この色を適用して」と再指示する。

色が苦手な人ほど、この順番の方が迷いにくいかもしれません。構成と色を同時に考えなくていいので。

どちらの順番でも、同じ .md が使えます。

## 作者自身のユースケース

デザイナーでもフロントエンドエンジニアでもなく、Figmaも使いません。やりたいことを LLM に投げてアプリを作るスタイルです。

この場合の色の問題はシンプルで、AIが毎回勝手に色を決めるとコンポーネントごとにバラバラになります。`.md` を渡して「このカラーシステムで作って」と一言添えるだけで、色の判断をAIに一任しつつ全体の一貫性が保てます。

「色が破綻しなければOK」という基準なら、このツールで十分です。

## おわりに

色を「何を選ぶか」より「どう選ぶか」として整理できると、迷う時間がずいぶん短くなります。

情景と時間帯を選んで、Hue を回して止める。このプロセス自体が、色を決める手がかりになると思っています。ぜひ試してみてください。
