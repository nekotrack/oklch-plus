## はじめに

Web ページを作るとき、色選びに時間をかけすぎた経験はないでしょうか。

RGB のスライダーを動かしても「なんとなくイメージと違う」、カラーピッカーで選んでも「背景と合わない」、そのままにしておくと全体がバラバラな印象になる。

色が苦手な理由のひとつは、**使っている色空間が人間の感覚と合っていない**からです。

---

## RGB・HSL ではなく OKLCH を使う理由

多くのカラーピッカーは RGB か HSL を使っています。

sRGB の Hex 値は直感的な決定が難しく、HSL の場合は L を 50% にしても、色によって明るさがバラバラに見えます。黄色と青を同じ L にしても、黄色の方がずっと明るく見えます。

一方、同じ LCH でも **OKLCH は直感とよく合います。**

```
L = Lightness（知覚的な明るさ）
C = Chroma（知覚的な鮮やかさ）
h = Hue（色の種類、0〜360°）
```

OKLCH の L を揃えると、色の種類が違っても同じ明るさに見えます。C を揃えると、同じ「鮮やかさ」に見えます。**「明るくしたい」「鮮やかにしたい」という言葉が、そのままスライダーの操作に対応します。**

---

## 作ったもの

**時彩（Ver. β4）**

情景（Hue）× 時間帯（L/C）の 2 軸選択から、AI・デザインツールに渡せるカラーシステムを自動生成するツールです。

## 使い方：3 ステップで完成

### Step 1：情景を選ぶ

6 つの情景から選ぶと、その情景の Hue（色相角）がベースになります。

| 情景 | Hue | 語 | 色の方向 |
|---|---|---|---|
| 茜・緋 | 25° | 茜に染まる | 赤橙 |
| 柿・麦 | 60° | 実りの季節 | 黄 |
| 柳・翠 | 160° | 翡翠の輝き | 緑 |
| 空・海 | 225° | 初夏の青 | 青 |
| 露・藍 | 264° | 藍に露 | 青紫 |
| 藤・宵 | 315° | 宵待ちの藤 | 紫 |

業種や用途ではなく、自然の情景から色の方向を選ぶ設計です。Hue 角度をボタンに表示しているので、「なぜこの色なのか」が数字でも確認できます。

### Step 2：時間帯を選ぶ

情景を選ぶとすぐ下に、選んだ Hue で描かれた 6 色のストリップが現れます。

| 時間帯 | L | C | 語 |
|---|---|---|---|
| 深夜 | 0.28 | 0.07 | 暗沈 |
| 夜明け | 0.42 | 0.14 | 薄明 |
| 朝 | 0.60 | 0.18 | 清澄 |
| 正午 | 0.68 | 0.15 | 白昼 |
| 夕暮れ | 0.55 | 0.19 | 夕映 |
| 宵 | 0.38 | 0.10 | 静寂 |

時間帯は明るさ（L）と鮮やかさ（C）に対応しています。夕暮れは最も彩度が高く、深夜は沈んだ暗い色になります。実際の色を見ながら選ぶことで、数値を意識せずに知覚的な明暗・彩度を選択できます。ストリップをタップすると即座にパレットが生成されます。

### Step 3：Hue を回して止める、書き出す

テーマカラーは Primary 1色で全体を統一する設計です。

Hue（色相）は複数の方法で動かせます。

| 方法 | 操作 |
|---|---|
| 自動回転 | `色相 H 回転` ボタン（30°/秒、回転中はボタン自体が色相を巡回） |
| スライダー | フッターの **h** スライダーを手動で動かす |
| 断面図ドラッグ | RESEARCH タブの L-C ガマット図上の現在値ポインターをドラッグ |
| 数値入力 | h の数値欄に 0〜360 を直接入力 |

:::message
`色相 H 回転` は想定外の使い方として発見しました。「選ぶ」ではなく**「色と出会う」**体験で、意外な色を見つけやすいです。単純に色が変わり続けるのを眺めているだけでも、なかなか楽しいです（個人の感想）。
:::

気に入った色で止めたら、現在値をコピーボタンで確定 →「md を書き出し」または「CSS をコピー」でカラーシステム定義ファイルが得られます。

## 生成されるもの

### カラーパレット（11色）

| 色 | 役割 |
|---|---|
| primary | メインボタン・リンク |
| primary-sub | ホバー時 |
| bg | ページ背景 |
| surface | カード背景 |
| surface-2 | サブ背景 |
| text | 本文テキスト |
| muted | 補足テキスト |
| border | 区切り線 |
| success / warning / error | 状態表示 |

すべて OKLCH 形式（CSS で直接使用可能）。

### フォント設定

`▶ フォント・サイズ設定を開く` から本文フォント（3択）・見出しフォント（3択）・サイズ（3択）を選べます。選択内容は見本欄（常時表示）にリアルタイム反映されます。

### md ファイル

```markdown
# Color System — 空・海 / 朝

## 1. プロジェクト概要
- 情景：空・海（初夏の青）h = 225°
- 時間帯：朝（清澄）L = 0.60 / C = 0.18
- 背景テーマ：Dark

## 2. CSS Variables（カラー）
:root {
  --color-primary:  oklch(0.600 0.180 220);
  ...
}

## 4. Typography
## 9. Agent Prompt Guide
```

## 2 つのワークフロー

色を渡すタイミングによって、2 つの進め方があります。どちらも同じ .md ファイルを使います。

### パターン A：色を先に決めてから渡す

ツールで情景・時間帯を選んで色を決める → md を書き出す → AI・デザインツールに渡す → デザイン・コード完成。

「ブランドカラーや世界観を起点にしたい」「色から全体のトーンを決めたい」という場合に向いています。

**AI への指示例：**

```
「空・海 / 朝」の配色（添付 color-system.md）を使って
（Webページ / Webアプリ / コンポーネント / SVGなど、目的を記入）を
デザインしてください。

色の役割は md の「使い分けガイド」を参照してください。

（追加の指示があればここに書き足してください）
```

括弧内は目的に応じて自由に書き換えてください。変数名で役割が明示されているので、Claude・ChatGPT・Gemini など、どの AI でも指示の書き方は変わりません。

### パターン B：骨格を先に作って色を後から流し込む

AI に「グレースケールで構成だけ作って」と依頼して骨格を完成させる → ツールで色を選ぶ → md を渡して「この色を適用して」と再指示。

色が苦手な人ほど、この順番の方が迷いにくいかもしれません。構成と色を同時に考えなくていいので。

```css
/* 骨格段階：変数名だけ決めておく */
.button { background: var(--color-primary); }
.card   { background: var(--color-surface); }

/* ツールで生成した :root を貼り付けるだけで全体に反映 */
:root {
  --color-primary: oklch(0.600 0.180 220);
  --color-surface: oklch(0.140 0.015 220);
}
```

チームで作業する場合も、構成担当と色担当で変数名の命名規則だけ合意しておけば並行作業が成立します。すでに完成したデザインの色だけ差し替えたいときも同様です。

## 使えるツール・用途

書き出した .md と CSS Variables は、以下のツールと用途で利用できます。

**Claude / Claude.ai**：.md をアップロードし、プロンプト例を貼り付けるだけ。

**ChatGPT / Gemini**：同じ .md を渡して同様に指示できます。

**Claude Code / GitHub Copilot / Cursor**：プロジェクトルートに .md を置き、`@color-system-sea-morning.md` で参照させるとカラーシステムを意識したコーディングを依頼できます。

**Figma**：OKLCH のネイティブサポートは未実装（2026年2月現在）ですが、OkColor などのプラグインで OKLCH を扱えます。

**コードに直接**：`:root` に貼り付けて `var(--color-primary)` で参照するだけです。oklch() は主要ブラウザで対応済み（2023年〜）。**Tailwind CSS v4 は OKLCH をネイティブ採用**しているため、設定ファイルにそのまま記述できます。`:root` を `@theme` に書き換えるだけで使えます。

用途はWebページに限りません。AIが md を読める形式であれば、Webアプリ・ダッシュボード・Reactコンポーネント・SVGグラフィックなど、指示が通ります。色の役割定義（セマンティクス）が入っているので、出力先を問わず同じ .md を使いまわせます。セマンティクスが用途に合わない場合は、AIに変数の読み替えや追加を依頼することもできます。

**プロトタイプや個人制作であれば1色構成のままで十分なケースが多く**、本番プロダクト向けには出発点として使い、必要に応じてパレットを拡張するという使い方が現実的です。仕上がりのイメージはツール内の見本欄で確認しながら進められます。

## 研究タブ：Oklch+ の可視化

「RESEARCH」タブでは色差研究の成果を可視化しています。

- **L-C ガマット（色域断面図）** — 現在の Hue における sRGB / P3 の形状。`色相 H 回転` と連動してリアルタイムに変化します
- **等間隔サンプリング比較** — OKLCH と Oklch+ で彩度を等間隔に並べると何が変わるか
- **知覚補正インジケーター** — Oklch+ による明度・彩度の変換量とガマット余裕

:::message
**研究タブを活用した操作**

① `色相 H 回転` で気に入った色相角のところで停止 → ② L-C ガマット図上の現在値ポインターをドラッグして明るさ・鮮やかさを選ぶ → ③ copy ボタンで現在値を確定 → ④ COLOR + PALETTE タブで適用
:::

**Oklch+** は Oklab 座標系に 3 パラメータの変換を加えることで CIEDE2000 を超えた色差モデルです。このタブの参考表示にのみ使用しており、AI ツールや CSS に渡されるパレットは標準の OKLCH 形式です。詳しくは第3弾の記事をご覧ください。

## おわりに

OKLCH は「明るく」「鮮やかに」「青寄りに」という言葉がそのままスライダーに対応する色空間です。情景と時間帯を選ぶことで、「空・海のような青、朝の清澄さで」という感覚のまま座標が決まります。

色決定先行でも骨格先行でも、同じ .md がデザインとコードをつなぐ橋になります。

ぜひ試してみてください。こんな使い方をしている、こんなツールと組み合わせた、という事例があればコメントで教えてください。どんなツールに育てるかは、使ってくださる方の声次第だと思っています。
