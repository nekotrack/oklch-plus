この記事では、OKLCH 補間の色かぶり問題に対する**連続的な hue 変調**の理論・実装・既存手法との比較を詳述します。本手法は、筆者が開発中の色差改善手法（便宜上 Oklch+ と呼ぶ・別途公開予定）の実装中に副産物として発見したものです。

---

## 1. 問題：極座標の特異点

OKLCH 補間は極座標（C, H）で行います。

```
h  = hA + t · Δh         ← 色相は線形に変化し続ける
C  = CA + t · (CB - CA)  ← 彩度は端点に向かってゼロへ
a  = C · cos(h)
b  = C · sin(h)
```

黒（C=0）は無彩色であり、色相が未定義の特異点です。
補間が黒端点に近づくにつれ C→0 になりますが、h は変化し続けます。
C が小さくても h が有効に機能するため、ab 平面で「原点を迂回する弧」が生じ、
その弧の途中が人間の目に茶色・紫として知覚されます。

この問題は以下の両方で発生します：

- 黒・白・グレーへの補間（端点が無彩色）
- 補間経路の中間で低彩度域を通過する経路（端点が有彩色でも起きる）

色空間の規格（sRGB・P3・将来規格）に依存せず、極座標構造の問題として普遍的に存在します。

---

## 2. 既存アプローチとその限界

この問題は CSS Color 4 仕様の策定時から認識されています。

**W3C CSS Working Group（Issue #6107, 2021 / Issue #9436, 2023）**

無彩色の hue を `NaN`（missing）として扱い、補間時に他方の端点の hue を継承する方法が採用されています。黒→赤の補間では黒の hue を赤の hue に揃える、という処理です。ただしこの方法は**二値的・不連続**であり、「どこから achromatic か」という閾値判定に依存するため、境界付近で hue が突然ジャンプするという問題が残ります。

**ColorAide 等のライブラリ**

無彩色に近い色を検出して hue を undefined に設定するアプローチです。現実的な近似として機能しますが、やはり離散的な検出ロジックに依存します。

**実務的なフォールバック**

OKLCH ツール・ライブラリの実装者コミュニティでは「片端が無彩色の場合は Oklab でグラデーションを行う」ことが事実上の標準的対処になっています（W3C Issue #9436 のコメント欄、および各種カラーライブラリのドキュメントで確認できます）。問題は確実に回避できますが、OKLCH の色相制御という利点を**丸ごと放棄**することになります。

いずれも「問題を回避する」アプローチであり、OKLCH の hue 制御を保ちながら連続的に解決する手法は提案されていません。

---

## 3. 解法：C に連動した NR 重み変調

### 着想

Oklab（デカルト座標 a, b の線形補間）では同問題が起きません。
そこで「C→0 付近では Oklab 線形補間に滑らかに退行し、C が大きい領域では Oklch 極座標補間を保つ」という**連続的なブレンド**を構成します。

### 重み関数

Oklch+（color metric として設計）の C 軸 NR パラメータ（n_C=0.87, σ_C=0.34）を流用します。
このパラメータは C≈0.34（中彩度）で感度が最大になるよう最適化されており、
「彩度が低いほど色相を信頼しない」という今回の目的とも方向性が一致しています。

```
w_h(C) = C^0.87 / (C^0.87 + 0.34^0.87)
```

| C の大きさ | w_h | 補間の性質 |
|------------|-----|-----------|
| C → 0      | → 0 | Oklab 線形（hue 寄与ゼロ） |
| C = σ = 0.34 | = 0.5 | 両者の中点 |
| C ≫ σ     | → 1 | Oklch 極座標（通常補間） |

NR（Naka-Rushton）関数の非線形性により、「C が中程度のときに hue 引力が最も効き、端点付近で自然にゼロへ」という、Power 関数や Log 関数では得られない曲線を実現しています。

### 実装

```javascript
function hueWeight(C, n = 0.87, sigma = 0.34) {
  if (C < 1e-12) return 0;
  const Cn = Math.pow(C, n), sn = Math.pow(sigma, n);
  return Cn / (Cn + sn);
}

// 補間内での適用（interpOkluch）
const wh    = hueWeight(C);
const a_lch = C * Math.cos(h);                    // Oklch 極座標
const b_lch = C * Math.sin(h);
const a_lab = labA[1] + t * (labB[1] - labA[1]); // Oklab 線形
const b_lab = labA[2] + t * (labB[2] - labA[2]);
const a = wh * a_lch + (1 - wh) * a_lab;
const b = wh * b_lch + (1 - wh) * b_lab;
```

追加パラメータはありません。

## 4. 幾何学的解釈

ab 平面上の補間経路で見ると：

- **Oklch（未変調）**: 原点を大きく迂回する弧 → 色かぶりの原因
- **Oklch+（NR変調）**: 原点を引力として内側に引き込まれた経路

「曲がった地図（色相環）を持っているとき、原点（無彩色）に近づくほど地図を信じるのをやめ、感覚（Oklab 線形）で歩く」という適応的な経路選択です。切り替えが NR により連続なので「いつ地図を捨てたか」が視覚的に現れません。

---

## 5. 観察された効果

`okluch_gradient_pub.html` の ab 平面補間経路で確認しています。

| プリセット | Oklch の問題 | Oklch+ での改善 |
|-----------|-------------|----------------|
| Vivid→Black | 茶色かぶり / ab 平面で大外回り | 解消 / 原点引力の内側経路 |
| Blue→Black | 紫かぶり / 半円状の迂回 | 解消 / ほぼ直線経路 |
| Blue→Yellow | 中間低 C 域での色かぶり | 意図せず改善 |
| Cyan→Red | 同上 | 意図せず改善 |
| Purple→Orange | 同上 | 意図せず改善 |

Blue→Yellow 等は端点が有彩色同士であるにもかかわらず改善しました。
これは「端点の問題」ではなく**「補間経路全体の C が低い領域における hue 安定化」**として、より広いスコープで機能することを示しています。

## 6. 設計上の判断

**素の OKLCH 補間（interpOklch）には意図的に適用していません。**
before/after の比較基準として、変調なしの挙動を保持するためです。W3C Issue への提出を念頭に置いた before/after の可視化です。

**n, σ の流用について。**
精密なフィッティングではなく「C→0 で滑らかにゼロへ」という定性的な挙動が目的であるため、Oklch+ の既存パラメータをそのまま使っています。追加パラメータを増やさないことでシステムの簡潔性を維持します。

## 7. 既存手法との比較

| アプローチ | 連続性 | hue 制御 | 実装コスト |
|-----------|--------|---------|-----------|
| W3C NaN 継承 | ❌ 二値 | △ 端点依存 | 低 |
| Oklab フォールバック | ✅ 連続 | ❌ 放棄 | 低 |
| **本提案（NR変調）** | **✅ 連続・微分可能** | **✅ 保持** | **低** |

連続的・微分可能な hue 変調関数を使った解法は、現時点で文献・仕様議論に見当たりません。

## おわりに

この手法は、暗所視シミュレーション（OklchSV・別途公開予定）の実装中に偶然発見したものです。意図的に低明度・低彩度領域を扱うシミュレーションを作っていたため、通常のデザイン用途では踏まれにくい問題が顕在化しました。

問題そのものは CSS Color 4 の策定議論（2021年〜）から認識されていましたが、「OKLCH の hue 制御を保ちながら連続的に解消する」というアプローチは提案されていませんでした。W3C の議論や Björn Ottosson（Oklab 作者）へのフィードバックとして持ち込む価値があると考えています。
