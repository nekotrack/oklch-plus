この記事では、OKLCH 補間の色かぶり問題に対する**連続的な hue 変調**の理論・実装・既存手法との比較を詳述します。本手法は、筆者が開発中の色差改善手法（便宜上 Oklch+ と呼ぶ・別途公開予定）の実装中に副産物として発見したものです。

---

## 1. 問題：極座標の特異点

OKLCH 補間は極座標（C, H）で行います。

```
h  = hA + t · Δh         ← 色相は線形に変化し続ける
C  = CA + t · (CB - CA)  ← 彩度は端点に向かってゼロへ
a  = C · cos(h)
b  = C · sin(h)
```

OKLCH の極座標構造から、2種類の色かぶり問題が発生します。

**①有彩色間で中間に低彩度域を通過する場合（モダンブラウザでも発生）**

```
青（H=262°）→ 黄（H=93°）の補間：
  中間でCが低下する領域を通過
  → h は変化し続け、ab 平面で迂回する弧が生じる
  → 中間に意図しない緑の帯が出現
```

**②黒・白・グレーへの補間（端点が無彩色）**

黒（C=0）は色相が未定義の特異点です。補間が黒端点に近づくにつれ C→0 になりますが、h は変化し続けます。ab 平面で「原点を迂回する弧」が生じ、茶色・紫として知覚されます。ただし後述するように、モダンブラウザは色相継承でこの問題を部分的に解消しています。

色空間の規格（sRGB・P3・将来規格）に依存せず、極座標構造の問題として普遍的に存在します。

---

## 2. 既存アプローチとその限界

この問題は CSS Color 4 仕様の策定時から認識されています。

**W3C CSS Working Group（Issue #6107, 2021 / Issue #9436, 2023）**

無彩色の hue を `NaN`（missing）として扱い、補間時に他方の端点の hue を継承する方法が採用されています。黒→赤の補間では黒の hue を赤の hue に揃える、という処理です。色相継承により、**有彩色→黒の補間は Oklab 線形補間と数学的に同一になります**。モダンブラウザではこの問題が実質的に解消されています。

ただし2つの限界があります。**①二値的・不連続**であること。「どこから achromatic か」という閾値判定に依存するため、境界付近で hue が突然ジャンプするという問題が残ります。**②有彩色間の補間には効果がない**こと。Blue→Yellow 等の経路で中間に低彩度域を通過する問題は、色相継承では解決できません。

**ColorAide 等のライブラリ**

無彩色に近い色を検出して hue を undefined に設定するアプローチです。現実的な近似として機能しますが、やはり離散的な検出ロジックに依存します。

**実務的なフォールバック**

OKLCH ツール・ライブラリの実装者コミュニティでは「片端が無彩色の場合は Oklab でグラデーションを行う」ことが事実上の標準的対処になっています（W3C Issue #9436 のコメント欄、および各種カラーライブラリのドキュメントで確認できます）。問題は確実に回避できますが、OKLCH の色相制御という利点を**丸ごと放棄**することになります。

いずれも「問題を回避する」アプローチであり、OKLCH の hue 制御を保ちながら連続的に解決する手法は提案されていません。

---

## 3. 解法：C に連動した NR 重み変調

### 前提：色相継承の仕様

本手法は**2つのケースに対して機能する**。

**ケース①：achromatic 端点への補間**
W3C CSS Color 4 と同様に、achromatic 端点（黒・白・グレー）の色相を有彩色側から継承する前提で動作する。色相継承により H が一定に保たれ C が低下するとき、NR 重みが C に応じて色相の影響をゼロへ収束させる。

**ケース②：有彩色間で低彩度域を通過する経路**
色相継承は適用されないが、補間経路中に C が低下する領域では同じ NR 重みが機能する。C が低いほど Oklab 線形に退行するため、色かぶりが抑制される。これがモダンブラウザでも残る問題を解消する本手法の主要な価値である。

### 着想

Oklab（デカルト座標 a, b の線形補間）では同問題が起きません。
そこで「C→0 付近では Oklab 線形補間に滑らかに退行し、C が大きい領域では Oklch 極座標補間を保つ」という**連続的なブレンド**を構成します。

### 重み関数

Oklch+（color metric として設計）の C 軸 NR パラメータ（n_C=0.87, σ_C=0.34）を流用します。
このパラメータは C≈0.34（中彩度）で感度が最大になるよう最適化されており、
「彩度が低いほど色相を信頼しない」という今回の目的とも方向性が一致しています。

```
w_h(C) = C^0.87 / (C^0.87 + 0.34^0.87)
```

| C の大きさ | w_h | 補間の性質 |
|------------|-----|-----------|
| C → 0      | → 0 | Oklab 線形（hue 寄与ゼロ） |
| C = σ = 0.34 | = 0.5 | 両者の中点 |
| C ≫ σ     | → 1 | Oklch 極座標（通常補間） |

NR（Naka-Rushton）関数の非線形性により、「C が中程度のときに hue 引力が最も効き、端点付近で自然にゼロへ」という、Power 関数や Log 関数では得られない曲線を実現しています。

### 実装

```javascript
function hueWeight(C, n = 0.87, sigma = 0.34) {
  if (C < 1e-12) return 0;
  const Cn = Math.pow(C, n), sn = Math.pow(sigma, n);
  return Cn / (Cn + sn);
}

// 補間内での適用（interpOkluch）
const wh    = hueWeight(C);
const a_lch = C * Math.cos(h);                    // Oklch 極座標
const b_lch = C * Math.sin(h);
const a_lab = labA[1] + t * (labB[1] - labA[1]); // Oklab 線形
const b_lab = labA[2] + t * (labB[2] - labA[2]);
const a = wh * a_lch + (1 - wh) * a_lab;
const b = wh * b_lch + (1 - wh) * b_lab;
```

追加パラメータはありません。

## 4. 幾何学的解釈

ab 平面上の補間経路で見ると：

- **Oklch（未変調）**: 原点を大きく迂回する弧 → 色かぶりの原因
- **Oklch+（NR変調）**: 原点を引力として内側に引き込まれた経路

「曲がった地図（色相環）を持っているとき、原点（無彩色）に近づくほど地図を信じるのをやめ、感覚（Oklab 線形）で歩く」という適応的な経路選択です。切り替えが NR により連続なので「いつ地図を捨てたか」が視覚的に現れません。

## 5. 観察された効果

`okluch_gradient_pub.html` の ab 平面補間経路で確認しています。

| プリセット | Oklch の問題 | Oklch+ での改善 | 発生環境 |
|-----------|-------------|----------------|---------|
| Blue→Yellow | 中間に緑の帯 / 低 C 域での迂回 | 解消 | モダンブラウザでも発生 |
| Cyan→Red | 中間に橙の帯 | 解消 | モダンブラウザでも発生 |
| Purple→Orange | 中間低 C 域での色かぶり | 解消 | モダンブラウザでも発生 |
| Vivid→Black | 茶色かぶり / ab 平面で大外回り | 解消 | 色相継承なし環境 |
| Blue→Black | 紫かぶり / 半円状の迂回 | 解消 | 色相継承なし環境 |

**有彩色間の補間（上3行）はモダンブラウザでも発生する、より現代的な問題です。**
これは「端点の問題」ではなく**「補間経路全体の C が低い領域における hue 安定化」**として機能しており、有彩色間でも achromatic 端点でも同一のメカニズムが働いています。

## 6. 設計上の判断

**素の OKLCH 補間（interpOklch）には意図的に適用していません。**
before/after の比較基準として、変調なしの挙動を保持するためです。W3C Issue への提出を念頭に置いた before/after の可視化です。

**n, σ の流用について。**
精密なフィッティングではなく「C→0 で滑らかにゼロへ」という定性的な挙動が目的であるため、Oklch+ の既存パラメータをそのまま使っています。追加パラメータを増やさないことでシステムの簡潔性を維持します。

## 7. 既存手法との比較

| アプローチ | 連続性 | hue 制御 | 微分可能 | 実装コスト |
|-----------|--------|---------|---------|-----------|
| W3C NaN 継承 | ❌ 二値 | △ 端点依存 | ❌ | 低 |
| Oklab フォールバック | ✅ 連続 | ❌ 放棄 | ✅ | 低 |
| **本提案（NR変調）** | **✅ 連続** | **✅ 保持** | **✅** | **低** |

### Oklab と Oklch の「良いとこ取り」

本手法の直感的な理解として、**Oklch+ の補間軌道は Oklab と Oklch の中間を通る**という点が重要です。

```
C が高い領域 : w_h → 1  →  Oklch 極座標（色相制御を最大化）
C が低い領域 : w_h → 0  →  Oklab 線形（安定性を最大化）
中間の領域   : 0 < w_h < 1  →  両者の加重平均
```

Oklab は色相制御を諦めることで安定性を得る。Oklch は色相制御を持つが低彩度域で不安定になる。Oklch+ はどちらかを選ぶのではなく、**彩度に応じて連続的に比重を変える**ことで両者の利点を統合しています。この「良いとこ取り」は、NR 重みの単調性によって数学的に保証されています。

### 微分可能であることの実用的な意味

```
w_h(C) = C^n / (C^n + σ^n)
```

この関数は C について**任意回微分可能**です。これは単に数学的な性質ではなく、実用上の利点があります：

- **アニメーション・トランジション** ：色相の切り替えが滑らかで、フレーム間に不連続点が生じない
- **GPU 並列処理との親和性** ：分岐なし・連続計算のため GLSL/WGSL 等への移植が容易
- **最適化への組み込み** ：勾配降下法を使うカラーシステム最適化に直接組み込める
- **W3C への提案適性** ：仕様として記述しやすく、実装が一意に定まる

連続的・微分可能な hue 変調関数を使った解法は、現時点で文献・仕様議論に見当たりません。
