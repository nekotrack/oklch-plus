OKLCH でグラデーションを作ると、黒への補間経路に茶色や紫の帯が現れることがあります。
この問題は W3C CSS Working Group でも 2021 年から議論されていますが、現在の CSS 仕様の解決策は完全ではありません。

この記事では問題の構造を整理し、「彩度に連動した連続的な hue 変調」という第三の解法を紹介します。
実装は JavaScript 数行で、追加パラメータなしに OKLCH の色相制御を保ったまま問題が解消できます。

---

## OKLCH の色は 3 軸で表される

```
L = 明度（0〜1）
C = 彩度（0〜0.4 程度）
H = 色相（0〜360°）
```

鮮やかな緑は `L=0.7, C=0.28, H=145°` のような値を持ちます。
HSL や RGB と違い、明度・彩度・色相が知覚的に均等なスケールで設計されているため、パレット生成やアクセシビリティ確保に急速に採用が広がっています。

ただし、構造的にひとつ問題を抱えています。

---

## 黒は「色相が未定義の特異点」

黒の値は次のようになります：

```
L=0, C=0, H=???
```

彩度ゼロは「色がない」ということなので、色相という概念自体が意味を持ちません。
数学的に言うと、黒は**色相が未定義の特異点（singular point）**です。

グラデーションを生成するとき、コンピューターは L・C・H をそれぞれ別々に補間します：

```
緑: L=0.7, C=0.28, H=145°
黒: L=0.0, C=0.00, H=???
```

L と C はゼロへ向かいますが、H は？目的地の色相が存在しないのに、補間は続きます。
H=145° を維持しながら C が絞られていくため、ab 平面上で**原点を迂回する弧**が生じます。
その弧の途中に彩度が中途半端に残った帯が見え、これが茶色・紫の正体です。

---

## 既存の解決策とその限界

**W3C の対処（CSS Color 4 仕様）**

無彩色の hue を `NaN`（missing）として扱い、補間時に有彩色側の hue を継承させる方法が採用されています。黒→赤の補間なら、黒の hue を赤の hue に揃える処理です。

限界は**二値的・不連続**であること。「どこから achromatic か」という閾値判定に依存するため、境界付近で hue が突然ジャンプする挙動が残ります。

**Oklab へのフォールバック**

カラーライブラリや実務では「片端が無彩色の場合は Oklab（デカルト座標）でグラデーションを作る」という方法が広く推奨されています。Oklab では色相が補間に介在しないので、問題そのものが起きません。

ただしこれは、**OKLCH の色相制御という利点を丸ごと諦める**ことを意味します。

---

## 解決策：彩度に連動した連続 hue 変調

考え方を変えます。

色相環は「色の地図」です。でも、彩度がゼロに近い領域では、その地図は意味を失います。
だとすれば：**彩度が高いときは色相環に従い、彩度が低くなるにつれて地図への信頼を減らしていけばいい。**

具体的には、各補間ステップの彩度 C に応じて、OKLCH 極座標補間と Oklab 線形補間をブレンドします：

```javascript
function hueWeight(C, n = 0.87, sigma = 0.34) {
  if (C < 1e-12) return 0;
  const Cn = Math.pow(C, n), sn = Math.pow(sigma, n);
  return Cn / (Cn + sn);  // C=0 → 0（Oklab線形）/ C大 → 1（Oklch極座標）
}

// 補間内での適用
const wh    = hueWeight(C);
const a_lch = C * Math.cos(h);
const b_lch = C * Math.sin(h);
const a_lab = labA[1] + t * (labB[1] - labA[1]); // Oklab 線形
const b_lab = labA[2] + t * (labB[2] - labA[2]);
const a = wh * a_lch + (1 - wh) * a_lab;
const b = wh * b_lch + (1 - wh) * b_lab;
```

ブレンドの割合には **Naka-Rushton 関数**（後述）を使います。
「中程度の彩度で最も効き、端点でゼロへ滑らかに収束する」という非線形の特性が、この用途にぴったり合っています。

ここで使用しているパラメータ（n=0.87、σ=0.34）は、筆者が開発中の色差改善手法 **Oklch+** の C 軸最適化から得られたものです。Oklch+ は OKLCH の彩度軸に Naka-Rushton 圧縮を適用することで知覚的均等性を改善するメトリックで、詳細は別途公開予定です。今回の Hue 変調は、その C 軸パラメータを補間経路の安定化に転用したものです。

---

## 幾何学的に見ると

ab 平面上の補間経路を可視化すると、違いが一目瞭然です：

- **Oklch（未変調）**: 原点を大きく迂回する弧 → 色かぶりの原因
- **Oklch+（NR変調）**: 原点を引力として内側に引き込まれた経路

「曲がった地図（色相環）を持っているとき、原点（無彩色）に近づくほど地図を信じるのをやめ、
感覚（Oklab 線形）で歩く」という適応的な経路選択です。切り替えが連続なので「いつ地図を捨てたか」が視覚的に現れません。

---

## 効果の確認

| プリセット | 変調なし Oklch | 変調あり Oklch+ |
|-----------|-------------|----------------|
| Vivid→Black | 茶色かぶり | 解消 |
| Blue→Black | 紫かぶり | 解消 |
| Blue→Yellow | 中間低彩度域での色かぶり | 意図せず改善 |
| Cyan→Red | 同上 | 同上 |

Blue→Yellow は端点が有彩色同士であるにもかかわらず改善しました。
「端点の問題」ではなく**「補間経路全体の低彩度領域における hue 安定化」**として機能しています。

| アプローチ | 連続性 | OKLCH hue 制御 |

| Oklab フォールバック | ✅ 連続 | ❌ 放棄 |

## 実務上の注意点

**影響を受けやすいケース**
- 鮮やかな色から黒・ダークグレーへのグラデーション
- ダークテーマのカラーパレット生成
- 暗所や夜のシーン表現

**影響を受けにくいケース**
- 有彩色同士（例: 青→オレンジ）のグラデーション
- 白や明るいグレーへの補間
- 彩度がもともと低いパレット

**CSS の `linear-gradient(in oklch ...)` との違い**
現在の CSS のグラデーション構文は、黒の色相を「もう片方の端点から継承する」という二値処理をしています。本手法は JavaScript ベースのカラーエンジンや独自補間実装で取り入れる余地があります。

## 補足：Naka-Rushton 関数とは

この解法で使った S 字カーブには名前と由来があります。

**中 研一（Ken-Ichi Naka）** と **William A.H. Rushton** ── 二人の生理学者が 1966 年に発表した研究に基づく関数です。テンチ（Tinca tinca・コイ科の淡水魚）の網膜を使った実験から光応答の特性を解明したもので、弱い光には敏感に反応し、強い光に対しては徐々に飽和していく様子を記述します。

「彩度が低くなるにつれて色相を信頼しなくなっていく」という感覚は、「光が弱くなるにつれて色の識別をやめる」という人間の視覚の性質とよく似ています。もともとは網膜の話ですが、思いがけない形で色の計算に役立ちました。
